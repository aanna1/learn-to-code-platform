## Why this matters

Every program you've written so far has the same problem: the moment you stop running it, everything it computed vanishes. The to-do list from Module 7 forgets every task. The dice game from Module 4 forgets your high score.

Memory is volatile. Disk is not. Take the data your program built up while running, write it somewhere persistent, and read it back next time the program starts. That one trick is the difference between a calculator you run once and software that gets more useful over time.

Files are also how programs communicate with the outside world. Grades come from a CSV someone exported from a spreadsheet. Configuration lives in a JSON file. Logs from a web server are text files you'll need to parse. By the end of this lesson, you'll have tools for all of them.

<Callout type="info">
On this platform, Python runs inside your browser. Files you write are stored in an in-memory filesystem — they work just like real files during a single run, but they reset when you press **Run** again. All the patterns here are identical to real Python; only the persistence lifetime is different. Press **Run** on any snippet to try it immediately.
</Callout>

## Core concepts

### The problem we're solving

Picture this: you run your to-do app, add "buy milk," "call mom," and "pay rent." You quit. You restart it. The list is empty. Those three strings lived inside a Python list, and Python lists live in memory. When the program exits, that memory goes back to the operating system.

For the list to survive, you need to write it to disk before quitting and read it back when starting. Files are the mechanism for doing that.

### Opening a file: the manual way, then the right way

Here's the most direct way to write a string to a file:

```python
f = open("greeting.txt", "w")
f.write("Hello from Python\n")
f.close()
```

Three steps. `open()` returns a file object. You call `.write()` to put data in, and `.close()` to flush it and release the file.

What happens if the program crashes between `open()` and `close()`? Or you forget the `close()` because you added an early `return`? The file might be left open and the data might not be flushed.

The fix is the `with` statement:

```python
with open("greeting.txt", "w") as f:
    f.write("Hello from Python\n")
```

Same effect, but `close()` is guaranteed. The `with` statement opens the file, binds it to `f` for the indented block, and closes it the moment that block ends — whether it exits normally or through an exception. You write the open; Python handles the close.

We'll always use `with`. There's no modern Python codebase where the bare form is preferable.

### Modes: read, write, append

The second argument to `open` is the mode. Three cover 95% of cases:

- `"r"` opens an existing file for reading. The file must exist or you get a `FileNotFoundError`. This is the default if you skip the mode argument.
- `"w"` opens a file for writing. If the file doesn't exist, it's created. If it already exists, **its contents are erased.**
- `"a"` opens a file for appending. New content goes at the end; existing content stays.

That second one bites everyone exactly once. Run the snippet below — it writes "Alice" to a file, then overwrites with "Bob," then reads it back:

<Runnable code={`# Write "Alice", then overwrite with "Bob"
with open("names.txt", "w") as f:
    f.write("Alice\\n")

with open("names.txt", "w") as f:   # "w" erases first!
    f.write("Bob\\n")

with open("names.txt", "r") as f:
    print(f.read())   # Only Bob — Alice is gone`} />

Every time mode `"w"` runs, the file is truncated to zero bytes before writing. Change the second `"w"` to `"a"` and both names survive.

A useful habit: if losing the existing file would ruin your day, look at the mode one more time before running the code.

### The newline you need to add yourself

Notice the `\n` at the end of `f.write("Alice\n")`. `f.write()` does not add a newline for you. If you wrote four names without `\n`, they'd land in the file as `AliceBobCarolDave`, one continuous string. `print()` adds newlines automatically; `write()` does not. This trips up nearly everyone the first time.

### A word on encoding

Text files store bytes. To turn those bytes back into characters, Python needs to know an encoding. Modern Python defaults to UTF-8, which handles every character you're likely to use. If you ever need to be explicit:

```python
with open("notes.txt", "r", encoding="utf-8") as f:
    contents = f.read()
```

This comes up when reading a file someone else made with a different encoding — a CSV exported from an older Windows Excel, for instance. The symptom is a `UnicodeDecodeError` halfway through reading. The fix is passing the right `encoding=` value, usually `"utf-8"` or `"cp1252"`.

### Reading a file three ways

Say you have a file with names in it:

```
Alice
Bob
Carol
```

You want those names in your Python program. There are three approaches, and the right one depends on how large the file is.

**Approach 1: read the whole thing as one string.**

```python
with open("names.txt", "r") as f:
    contents = f.read()
print(contents)
```

`contents` is now a single string: `"Alice\nBob\nCarol\n"`. Fine for small files, terrible for a 10-gigabyte log file (you'd exhaust memory).

**Approach 2: read all lines into a list.**

```python
with open("names.txt", "r") as f:
    lines = f.readlines()
print(lines)
```

`lines` is `['Alice\n', 'Bob\n', 'Carol\n']`. The `\n` stays at the end of each entry. Strip it when you don't want it:

```python
with open("names.txt", "r") as f:
    lines = [line.strip() for line in f.readlines()]
print(lines)  # ['Alice', 'Bob', 'Carol']
```

**Approach 3: iterate line by line.**

```python
with open("names.txt", "r") as f:
    for line in f:
        print(line.strip())
```

Default to this one. It reads one line at a time, works on files of any size, and never loads the whole file into memory at once.

A common bug: forgetting `.strip()` when printing. If your output has extra blank lines between every name, you're double-printing newlines — once from the file's `\n`, once from `print()`. Strip first, or pass `end=""` to `print`.

What does this print? Think before running:

<Runnable code={`# Set up a file first
with open("names.txt", "w") as f:
    f.write("Alice\\nBob\\nCarol\\n")

# Now read it with readline() calls
with open("names.txt", "r") as f:
    first_line = f.readline()
    second_line = f.readline()
print(first_line)
print(second_line.strip())`} />

<Callout type="tip">
`first_line` is `"Alice\n"` — printing it gives `Alice` plus the string's newline, then `print` adds its own newline on top, so you see `Alice` followed by a blank line. `second_line.strip()` removes the `\n`, so you just see `Bob`. This is why so many file-reading examples look like they have phantom blank lines: forgotten strips.
</Callout>

### CSV: when commas have meaning

A lot of real-world data lives in CSV files. Commas separate columns; newlines separate rows. The first row is usually a header.

```
name,grade
Alice,92
Bob,77
Carol,88
```

Your first instinct will be to read each line and split on the comma:

```python
with open("students.csv", "r") as f:
    for line in f:
        parts = line.strip().split(",")
        print(parts)
```

That works until it doesn't. Consider this row:

```
"Smith, John",88
```

A name with a comma inside it, wrapped in quotes. `line.split(",")` sees commas, gives you three pieces, and the name is destroyed. The `csv` module knows the quoting rules. Use it:

<Runnable code={`import csv

# Write a sample CSV
with open("students.csv", "w", newline="") as f:
    writer = csv.writer(f)
    writer.writerows([
        ["name", "grade"],
        ["Alice", 92],
        ["Bob", 77],
        ['"Smith, John"', 88],
    ])

# Read it back
with open("students.csv", "r") as f:
    reader = csv.reader(f)
    for row in reader:
        print(row)`} />

`csv.reader` iterates over the file and gives you each row as a list of fields, with all quoting and escaping handled.

Two things worth noticing. First, the header row comes through like any other row — you'll usually `next(reader)` to skip it, or use `DictReader` instead. Second, every value is a string. Even `92`. Cast with `int()` or `float()` when you need to do math.

`csv.DictReader` is friendlier when the file has headers:

<Runnable code={`import csv

# Write the CSV
with open("students.csv", "w", newline="") as f:
    writer = csv.writer(f)
    writer.writerows([["name", "grade"], ["Alice", 92], ["Bob", 77], ["Carol", 88]])

# DictReader gives each row as a dictionary
with open("students.csv", "r") as f:
    reader = csv.DictReader(f)
    for row in reader:
        print(row["name"], int(row["grade"]))`} />

Each row arrives as a dictionary keyed by the header names. No counting commas, no remembering that "name was column zero." If someone reorders the columns later, your code still works.

**Writing CSV.** Use `csv.writer`. Pass `newline=""` to `open()`, or on Windows you'll get a blank line between every row:

```python
import csv

rows = [["name", "grade"], ["Alice", 92], ["Bob", 77]]
with open("students_out.csv", "w", newline="") as f:
    writer = csv.writer(f)
    writer.writerows(rows)
```

`writerow` writes one row; `writerows` writes a list of them.

Now try it yourself: read the CSV below with `DictReader`, cast each grade to an integer, and print the average. Aim for under 10 lines:

<Runnable code={`import csv

# Setup: write the CSV
with open("students.csv", "w", newline="") as f:
    csv.writer(f).writerows([["name","grade"],["Alice","92"],["Bob","77"],["Carol","88"]])

# Your turn: read it with DictReader, compute the average grade
# Hint: build a list of int grades, then sum / len`} />

<Callout type="tip">
One clean approach: `grades = [int(row["grade"]) for row in csv.DictReader(f)]`, then `sum(grades) / len(grades)`. A list comprehension over `DictReader` pulls just the grades while casting each to `int`. Then `sum / len` for the average.
</Callout>

### JSON: structure that survives

CSV is great for flat rectangular data: rows and columns. JSON is what you reach for when your data has shape — lists inside dicts, dicts inside lists, anything nested.

The two functions you need: `json.dump` writes a Python object to a file, and `json.load` reads it back.

<Runnable code={`import json

todos = [
    {"title": "buy milk", "done": False},
    {"title": "call mom", "done": True},
]

with open("todos.json", "w") as f:
    json.dump(todos, f, indent=2)

# Read it back
with open("todos.json", "r") as f:
    loaded = json.load(f)

print(loaded[0]["title"])   # buy milk
print(loaded[1]["done"])    # True`} />

`todos` comes back as a list of dictionaries with exactly the shape it had when you wrote it. This is the standard way to persist a Python data structure: dump it on the way out, load it on the way in.

The `indent=2` argument makes the file human-readable. Skip it and you get one long line — the computer reads it just as well, but you can't.

JSON is strict about types. Python `True` becomes JSON `true`, `None` becomes `null`, and only basic types survive the round trip: strings, numbers, lists, dicts, booleans, `None`. A Python `datetime` object can't be JSON-encoded without extra work, and custom classes need to be converted to plain dicts first.

The pattern for persistent data is always **load → mutate → save**:

<Runnable code={`import json
from pathlib import Path

FILE = Path("favorites.json")

# Load existing list, or start with an empty one
favorites = json.loads(FILE.read_text()) if FILE.exists() else []

# Add a new entry
favorites.append("Blue")   # in a real program: input("Favorite: ")

# Save the updated list
FILE.write_text(json.dumps(favorites, indent=2))
print(favorites)`} />

<Callout type="tip">
The `if FILE.exists() else []` pattern is how every persistent program in this curriculum starts. The first run creates a fresh list; every later run loads and extends the existing one.
</Callout>

### File paths: stop concatenating strings

Where does `open("notes.txt")` look for the file? In the current working directory — wherever Python was when the program started. The instinct to build paths with string concatenation breaks on Windows (backslash vs forward slash) and in subtle ways you won't catch until the code runs on a different machine.

`pathlib` is the fix. A `Path` object knows how to combine pieces correctly on any OS:

```python
from pathlib import Path

folder = Path("/home/user/projects/data")
path = folder / "notes.txt"   # the / operator builds paths safely
```

`Path` also has methods that save you the open/read/close dance for simple one-shot cases:

<Runnable code={`from pathlib import Path

p = Path("notes.txt")
p.write_text("Hello from pathlib\\n")  # write and close in one call

contents = p.read_text()               # read and close in one call
print(contents)

print(p.exists())   # True`} />

For line-by-line or large-file reading, stick with `with open(...)`. For quick reads and writes, `pathlib` is shorter.

A handy idiom for scripts that need to find files next to themselves, regardless of where they're run from:

```python
from pathlib import Path

CONFIG = Path(__file__).parent / "config.json"
```

`__file__` is the path of the running script; `.parent` is the folder containing it. This fixes the "works from the project root, breaks from anywhere else" bug.

## Common pitfalls

**Using `"w"` when you meant `"a"`.** Every run erases the previous content. For logs, journals, and any additive work, use `"a"`.

**Forgetting `.strip()` on lines read from a file.** The line is `"Alice\n"`. The comparison `line == "Alice"` always fails. Strip first.

**Forgetting that CSV values are strings.** `total += row[1]` raises `TypeError`. Cast with `int(row[1])` before doing math.

**CSV writer producing blank lines on Windows.** Pass `newline=""` to `open()`. Without it, Python's default newline translation fights with the `csv` module and you get an empty row after every real one.

**Relative path confusion.** `open("config.json")` works from the project root and fails from everywhere else. Use `Path(__file__).parent / "config.json"` for files that must be found next to the script.

## Recap

- `with open(path, mode) as f:` is the right way to open a file. The file closes when the block ends, even on exceptions.
- Modes: `"r"` reads, `"w"` writes and erases, `"a"` appends. Pick wrong and you lose data.
- Three ways to read: `.read()` for the whole file, `.readlines()` for a list of lines, or `for line in f:` for one line at a time. Default to the third.
- Lines from a file end in `\n`. Call `.strip()` before comparing or printing them.
- `csv.reader` and `csv.DictReader` handle quoted commas and other CSV quirks. CSV values come in as strings; cast them.
- `json.dump` and `json.load` round-trip Python lists and dicts through JSON files. Use `indent=2` for readability.
- `pathlib.Path` builds paths that work on any OS and offers `.read_text()` / `.write_text()` for one-shot work.
