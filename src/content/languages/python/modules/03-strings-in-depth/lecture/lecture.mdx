## Why this matters

Roughly everything that crosses the boundary of your program — user input, lines from files, JSON from APIs, command-line arguments, log entries — arrives as a string. Module 2 dealt with the special case where what you actually want is a number, so you convert as fast as you can. Module 3 is the much bigger category: when you want the string itself, and you need to do something with it.

What "something" usually means: clean it up, pull part of it out, check whether it matches a pattern, glue it to other strings. Concrete examples you'll meet within a week of using Python in real life:

- A user typed `"  Ada Lovelace  "` with extra whitespace because they're on a phone. You need to strip it before storing.
- You have an email `"user@example.com"` and you need just the domain.
- A CSV line `"name,age,city,occupation"` needs to break into four pieces.
- A timestamp `"2024-11-15T14:30:00Z"` has a date and a time stuck together, and you need them apart.

What goes wrong without this: you can't tell the difference between `" Ada"` and `"Ada"` when comparing, so a login fails for no reason. You call `s.lower()` and wonder why the original is still capitalized. You try `s[0]` on a name field and crash when somebody hits Enter without typing anything. None of these are exotic. They're all going to happen, multiple times, this month.

## Core concepts

### Strings are sequences of characters

A string in Python isn't an atomic blob of text. It's a *sequence* — an ordered collection of individual characters, each at a numbered position. Same way an array works, if you've seen one in another language.

```python
word = "Python"
```

That string has six characters: `P`, `y`, `t`, `h`, `o`, `n`. Each lives at a numbered slot, starting at zero:

```
position:   0   1   2   3   4   5
character:  P   y   t   h   o   n
```

Yes, zero. Every introductory programmer fights this for about a week. The first character is at position `0`, not `1`. The reasons are partly historical and partly mathematical. The point is to internalize it.

You grab a single character with square brackets:

<Runnable code={`word = "Python"
print(word[0])   # P
print(word[1])   # y
print(word[5])   # n`} />

You can also count backwards from the end with negative indices:

<Runnable code={`word = "Python"
print(word[-1])   # n — last character
print(word[-2])   # o — second to last`} />

<Callout type="tip">
`word[-1]` is the most useful trick in this section. Need the last character of any string, no matter how long? `s[-1]`. You don't need to compute `len(s) - 1`.
</Callout>

### What if you go past the end?

What happens if you ask for `word[10]` when the word is only six characters?

<Runnable code={`word = "Python"
print(word[10])`} />

<Callout type="tip">
You get an `IndexError: string index out of range`. Python's standard "you asked for a slot that doesn't exist" error. The same applies to negative indices: `word[-100]` is also an error. There's no automatic wrap-around or default.

This is a real bug worth keeping in mind. If a string might be empty — and an `input()` call could always return `""` — then `s[0]` crashes. For now, just remember the failure mode. Module 4's conditionals will give you the tools to guard against it.
</Callout>

### Strings are immutable

You cannot change a character inside a string in place.

<Runnable code={`word = "hello"
word[0] = "J"`} />

<Callout type="tip">
That gives `TypeError: 'str' object does not support item assignment`. Strings in Python are *immutable* — once created, the character sequence can't be modified. Every operation that looks like it "changes" a string actually produces a new string and leaves the original alone.
</Callout>

This is the source of one of the most common new-programmer bugs in Python:

<Runnable code={`name = "  Ada  "
name.strip()        # returns "Ada" but throws it away
print(name)         # still "  Ada  ", spaces and all`} />

`.strip()` did its job — it produced a new string with the whitespace removed — but we didn't capture the result, so it vanished. Fix it by grabbing the return value:

<Runnable code={`name = "  Ada  "
name = name.strip()
print(name)         # "Ada"`} />

The same pattern applies to every string method. They all return new strings; none modify the original.

### Slicing

Slicing is how you grab a *range* of characters instead of just one. The syntax is `s[start:stop]`:

<Runnable code={`word = "Python"
print(word[0:3])    # "Pyt"
print(word[1:4])    # "yth"
print(word[2:6])    # "thon"`} />

Notice that `word[0:3]` gives `"Pyt"` — three characters, not four. The `stop` value is *exclusive*. The slice includes everything from `start` up to but not including `stop`. So `[0:3]` covers positions 0, 1, 2.

If you want N characters starting from position `start`, the stop is `start + N`, not `start + N - 1`. Eventually this becomes reflex. Until it does, the half-open convention trips everyone up.

You can leave either side blank, and Python fills in defaults — start of string for an empty `start`, end of string for an empty `stop`:

<Runnable code={`word = "Python"
print(word[:3])     # "Pyt"     — first three
print(word[3:])     # "hon"     — from position 3 to the end
print(word[:])      # "Python"  — a full copy`} />

Negative indices work in slices too:

<Runnable code={`word = "Python"
print(word[-3:])    # "hon"    — last three characters
print(word[:-1])    # "Pytho"  — everything except the last character`} />

And the third slice parameter is a step:

<Runnable code={`word = "Python"
print(word[::2])    # "Pto"     — every second character
print(word[::-1])   # "nohtyP"  — reversed`} />

`[::-1]` says: start at the default (end, because the step is negative), stop at the default (beginning), step by -1. It walks the string from end to beginning — the most common Pythonic way to reverse a string.

Slices, unlike single-index lookups, don't raise errors when you go out of bounds:

```python
print(word[0:100])    # "Python"   — silently clipped
print(word[100:200])  # ""         — empty string, no error
```

That's friendlier than indexing, but slightly more dangerous: a slice that returns `""` looks like it worked, so you may not notice the bug.

**Predict before you run.** What will each line print? Make a guess, then press Run:

<Runnable code={`s = "automation"
print(s[0])
print(s[-1])
print(s[0:4])
print(s[4:])
print(s[::-1])
print(s[2:-2])`} />

<Callout type="tip">
- `s[0]` → first character: `a`.
- `s[-1]` → last character: `n`.
- `s[0:4]` → positions 0–3: `auto`.
- `s[4:]` → position 4 to the end: `mation`.
- `s[::-1]` → reversed: `noitamotua`.
- `s[2:-2]` → position 2 up to (not including) position -2, which is index 8: positions 2–7 → `tomati`. Mixing positive and negative bounds is legal and sometimes very useful.
</Callout>

### String methods: the core dozen

A *method* is a function that belongs to an object and is called with a dot: `s.method()`. Strings come with dozens of methods built in. You need about a dozen well enough to recognise when one would help.

Every method returns a *new* string. None modify the original.

#### Case conversion: `.lower()`, `.upper()`, `.title()`

<Runnable code={`greeting = "Hello, World"
print(greeting.lower())    # "hello, world"
print(greeting.upper())    # "HELLO, WORLD"
print(greeting.title())    # "Hello, World" — first letter of each word
print(greeting)            # "Hello, World" — unchanged`} />

These are essential for case-insensitive comparison. To check if the user typed "yes" without caring about capitalisation:

```python
answer = input("Continue? ").lower()
if answer == "yes":
    print("Continuing.")
```

Note `.lower()` chained right onto the `input()` call. That's idiomatic: clean the input the moment you receive it, before you store it.

#### Whitespace cleanup: `.strip()`, `.lstrip()`, `.rstrip()`

`.strip()` removes whitespace (spaces, tabs, newlines) from both ends. `.lstrip()` does the left side only; `.rstrip()` the right.

<Runnable code={`raw = "   Ada Lovelace\n"
print(raw.strip())     # "Ada Lovelace"`} />

Anyone who has read user input or a line from a file has hit a "why doesn't this match" bug caused by trailing whitespace. Use `.strip()` on user input by default unless you have a specific reason not to.

You can pass `.strip()` a string of characters to strip instead of whitespace:

<Runnable code={`url = "https://example.com/"
print(url.rstrip("/"))  # "https://example.com"`} />

#### Finding and counting: `.find()`, `.count()`, `.startswith()`, `.endswith()`, `in`

<Runnable code={`sentence = "the quick brown fox jumps over the lazy dog"
print(sentence.find("fox"))        # 16 — position where "fox" starts
print(sentence.find("zebra"))      # -1 — not found
print(sentence.count("the"))       # 2
print(sentence.startswith("the"))  # True
print(sentence.endswith("dog"))    # True
print("fox" in sentence)           # True`} />

<Callout type="warning">
`.find()` returns `-1` when nothing is found — not `None`, not an error. So if you slice with the result without checking:

```python
position = sentence.find("zebra")
print(sentence[position:position + 5])   # silent garbage
```

You'll get weird output, because `sentence[-1:4]` is a valid but bizarre slice. Always check `if position != -1:` before slicing with it, or use the `in` operator when you only care whether something exists.
</Callout>

#### Replacing and splitting: `.replace()`, `.split()`, `.join()`

`.replace(old, new)` returns a new string with every occurrence of `old` swapped out:

<Runnable code={`text = "I like cats. Cats are great."
print(text.replace("cats", "dogs"))    # "I like dogs. Cats are great."
print(text.replace("Cats", "Dogs"))    # "I like cats. Dogs are great."`} />

Case sensitive — `"cats"` and `"Cats"` are different targets.

`.split()` breaks a string into a list of pieces. With no argument, it splits on any whitespace:

<Runnable code={`sentence = "the quick brown fox"
parts = sentence.split()
print(parts)    # ['the', 'quick', 'brown', 'fox']`} />

You can pass a separator:

<Runnable code={`csv_line = "Ada,28,Engineer,Austin"
fields = csv_line.split(",")
print(fields)   # ['Ada', '28', 'Engineer', 'Austin']`} />

And cap the number of splits with a second argument:

<Runnable code={`full_name = "Mary Anne Smith"
first, last = full_name.split(" ", 1)
print(first)    # "Mary"
print(last)     # "Anne Smith"`} />

That `1` says "split at most once." Without it, `split(" ")` gives three pieces, and `first, last = ...` would raise a `ValueError` because three things don't fit in two slots.

`.join()` is `.split()` in reverse. It takes a sequence of strings and glues them with a separator:

<Runnable code={`parts = ["the", "quick", "brown", "fox"]
sentence = " ".join(parts)
print(sentence)   # "the quick brown fox"

csv_row = ",".join(["Ada", "28", "Engineer", "Austin"])
print(csv_row)    # "Ada,28,Engineer,Austin"`} />

The syntax reads backwards at first: the *separator* is the string you call `.join()` on, and the *pieces* are the argument. `" ".join(parts)` means "use a space to join up the elements of parts."

### Concatenation: `+`, f-strings, and `.join()`

You have three main ways to combine strings, and they're not interchangeable.

**The `+` operator** works between two strings:

<Runnable code={`first = "Ada"
last = "Lovelace"
full = first + " " + last
print(full)    # "Ada Lovelace"`} />

The trap: `+` only works *between strings*. Mix a string and a number and you crash:

<Runnable code={`age = 28
print("I am " + age + " years old.")`} />

<Callout type="tip">
Fix by converting the number with `str()`, or — better — use an f-string: `print(f"I am {age} years old.")`. f-strings don't require explicit `str()` calls and are easier to read.
</Callout>

**f-strings** are usually the right answer when mixing text and variables. Use them by default.

**`.join()`** is the right answer when you have a *list* of strings (often built dynamically) and want one separator between each. Combining names where a middle name is sometimes missing? `.join()`. Building a comma-separated CSV row? `.join()`. Concatenating ten individually named variables? f-string.

### Escape sequences

Sometimes you need a character that's hard to type literally — either because it's invisible (a newline) or because it would confuse Python's parser (a quote inside a quoted string). Escape sequences are how you get them in: a backslash followed by another character.

<Runnable code={`print("Line 1\nLine 2")     # newline between them
print("Column1\tColumn2")   # tab between them
print("She said \"hi\"")    # literal double quote
print("Path: C:\\Users")    # literal backslash`} />

The common ones:

- `\n` — newline
- `\t` — tab
- `\"` — literal double quote (only needed inside `"..."`)
- `\'` — literal single quote (only needed inside `'...'`)
- `\\` — literal backslash

That last one matters more than it looks. A single `\` starts an escape sequence. If you want an actual backslash, you write `\\`. Forgetting this is a constant source of bugs with Windows file paths:

<Runnable code={`path = "C:\new_folder\test.txt"   # broken: \n is a newline, \t is a tab
print(path)`} />

<Callout type="tip">
Either escape the backslashes (`"C:\\new_folder\\test.txt"`), use forward slashes (which Python and Windows both accept in `open()`), or use a *raw string*:

```python
path = r"C:\new_folder\test.txt"
```

The `r` prefix tells Python "don't interpret escape sequences — treat the characters literally." Raw strings come back in Module 12 (regular expressions) where they're essential.
</Callout>

### Format specifiers: alignment, width, decimals

Module 2 introduced `f"{price:.2f}"` for two decimal places. Format specifiers can do more: width, alignment, and padding character.

The full syntax inside the braces is:

```
{value:[fill][align][width][.precision][type]}
```

In practice you'll only use a couple of pieces at a time:

<Runnable code={`name = "Ada"
print(f"{name:<10}|")    # "Ada       |" — left-align in width 10
print(f"{name:>10}|")    # "       Ada|" — right-align in width 10
print(f"{name:^10}|")    # "   Ada    |" — center-align in width 10`} />

The `<`, `>`, and `^` characters mean left-align, right-align, and center-align. The number after them is the field width. The `|` is just a literal character so you can see where the field ends.

Putting it together to build a tidy receipt:

<Runnable code={`print(f"{'item':<20}{'price':>10}")
print(f"{'apples':<20}{2.49:>10.2f}")
print(f"{'bananas':<20}{0.99:>10.2f}")
print(f"{'total':<20}{3.48:>10.2f}")`} />

The left column is 20 characters wide, left-aligned. The right column is 10 wide, right-aligned, formatted as a float with 2 decimal places.

You can pad with a specific character instead of a space:

<Runnable code={`print(f"{42:0>5}")      # "00042" — pad with zeros, right-aligned, width 5
print(f"{42:->5}")      # "---42" — pad with dashes, right-aligned, width 5`} />

The zero-pad version is common when formatting numbers as IDs or file names: `report_00042.csv`.

**Predict before you run.** Given an email like `"  Ada.Lovelace@Example.COM  "`, what will the code below produce?

<Runnable code={`email = "  Ada.Lovelace@Example.COM  "
clean = email.strip().lower()
username, domain = clean.split("@")
print(username)
print(domain)`} />

<Callout type="tip">
`email.strip().lower()` chains two methods: strip the whitespace, then lowercase the result. `.split("@")` on `"ada.lovelace@example.com"` gives a two-element list, which unpacks into `username` and `domain` in one line. Result: `ada.lovelace` and `example.com`.
</Callout>

## Common pitfalls

**1. Treating string methods as if they modified in place.**

```python
name = "ada"
name.upper()
print(name)   # "ada" — unchanged
```

`.upper()` returned a new string that was immediately discarded. Capture it: `name = name.upper()` or use it inline: `print(name.upper())`.

**2. Forgetting that slice `stop` is exclusive.**

```python
word = "Python"
print(word[0:4])    # "Pyth" — 4 characters, positions 0 through 3
```

If you want N characters starting at position `start`, the stop is `start + N`, not `start + N - 1`.

**3. Indexing past the end of a string.**

```python
name = input("Name: ")    # user just hits enter → ""
first_letter = name[0]
# IndexError: string index out of range
```

Always assume `input()` can return `""`. Check `if name:` (Module 4) before indexing or slicing.

**4. Forgetting `()` when calling a method.**

```python
name = "ada"
print(name.upper)     # prints the method object, not the result
print(name.upper())   # "ADA"
```

A method without parentheses is like a phone number you wrote down but never dialed.

**5. Using `+` between a string and a number.**

```python
age = 30
print("Age: " + age)
# TypeError: can only concatenate str (not "int") to str
```

Convert with `str(age)`, or use an f-string: `print(f"Age: {age}")`.

**6. Treating `.find()`'s `-1` return as a valid position.**

```python
text = "hello"
position = text.find("z")          # -1
print(text[position:position+3])   # "o" — almost certainly not what you want
```

Either check `if position != -1:` (Module 4) or use `if "z" in text:` when you only care about presence.

## Recap

- Strings are zero-indexed sequences. `s[0]` is the first character; `s[-1]` is the last.
- Slicing is `s[start:stop:step]`. `stop` is exclusive. Missing values default to the string's start and end. A negative step walks backward; `s[::-1]` reverses.
- Strings are immutable. Every method returns a new string and leaves the original alone. Capture the return value or it's gone.
- The core methods worth knowing reflexively: `.lower()`, `.upper()`, `.strip()`, `.replace()`, `.split()`, `.join()`, `.find()`, `.count()`, `.startswith()`, `.endswith()`, and the `in` operator.
- Three ways to combine strings: `+` for two literals, f-strings for mixing text and variables, `.join()` for sequences of pieces.
- Escape sequences (`\n`, `\t`, `\\`, `\"`) embed special characters. Raw strings (`r"..."`) turn escapes off — useful for file paths and regex.
- Format specifiers (`<`, `>`, `^` for alignment; a number for width; `.2f` for decimal places) let f-strings produce tidy aligned output.
