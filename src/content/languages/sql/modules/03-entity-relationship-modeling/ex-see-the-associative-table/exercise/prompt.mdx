In the lecture, an employee could work on many projects and a project could have many employees — a **many-to-many** relationship. You saw that it can't be stored with a single foreign key, so it's resolved into an **associative entity**: a table that sits between the two, where each row records exactly one pairing.

Here that table is `assignment`. The schema above sets up three tables — `employee`, `project`, and the `assignment` table that links them. The starter query shows you the projects, but a project row has nowhere to list everyone working on it. The pairings live in the associative entity, and that's where you need to look.
